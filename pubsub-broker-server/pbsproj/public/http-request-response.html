HTTP Request/Response Anatomy

Your pub-sub broker communicates entirely through HTTP/1.1 messages. Every byte you read from the socket and every byte you write back must conform to the format described here. This page is the single most important reference for the project.

The Structure of an HTTP Request

An HTTP request has exactly four parts, in order:

  1. Request line -- the method, path, and HTTP version
  2. Headers -- zero or more key-value pairs
  3. Blank line -- signals the end of the headers
  4. Body -- optional, only present when Content-Length > 0

Request Line

The first line of every request follows this format:

METHOD /path HTTP/1.1\r\n

The three parts are separated by single spaces. The line ends with \r\n (a carriage return followed by a newline, known as CRLF).

Examples from the pub-sub API:

GET /status HTTP/1.1\r\n GET /subscribe/weather HTTP/1.1\r\n POST /publish/weather HTTP/1.1\r\n

Headers

Each header is a line in the format Name: value\r\n. There is a colon and a space between the name and value. Header names are case-insensitive, so Content-Length, content-length, and CONTENT-LENGTH all mean the same thing.

Host: localhost:8080\r\n Content-Type: application/json\r\n Content-Length: 42\r\n

Blank Line (End of Headers)

After the last header, you send one more \r\n. Since the header itself already ended with \r\n, you will see \r\n\r\n in the byte stream. This double-CRLF is how you know the headers are finished.

Body

If the request includes a Content-Length header, the body follows immediately after the blank line. You must read exactly Content-Length bytes -- no more, no less.

Full Request Example

A client publishing a weather update:

POST /publish/weather HTTP/1.1\r\n Host: localhost:8080\r\n Content-Type: application/json\r\n Content-Length: 28\r\n \r\n {"temp": 72, "sky": "clear"}

Count the body: {"temp": 72, "sky": "clear"} is exactly 28 bytes. The Content-Length header told us to read exactly that many bytes after the blank line.

Tip: When parsing the request line, split on spaces. The first token is the method, the second is the path, and the third is the version string. You can ignore the version for this project, but you must still read past it.

The Structure of an HTTP Response

A response has the same four parts, but the first line is different:

  1. Status line -- the HTTP version, status code, and reason phrase
  2. Headers
  3. Blank line
  4. Body

Status Line

HTTP/1.1 200 OK\r\n

Common status codes you will use:

  • 200 OK -- success
  • 201 Created -- resource created (e.g., new subscription)
  • 404 Not Found -- unknown path
  • 400 Bad Request -- malformed request
  • 405 Method Not Allowed -- wrong method for this path

Full Response Example

Responding to GET /status:

HTTP/1.1 200 OK\r\n Content-Type: application/json\r\n Content-Length: 45\r\n \r\n {"subscribers": 3, "topics": ["weather","news"]}

Understanding CRLF

HTTP requires \r\n (carriage return + line feed) as the line terminator. In Java, that is the string "\r\n". Do not use just \n -- many HTTP clients and test tools will reject responses that use bare newlines.

When you read from the socket, you can read line-by-line until you see an empty line (one that is just \r\n). In Java, if you use a BufferedReader, readLine() strips the line terminator, so an empty line comes back as an empty string "".

Tip: Be careful with BufferedReader when reading the body. Since the body may contain binary data or may not end with a newline, you should use read(char[], offset, length) to read exactly Content-Length characters after the blank line.

How Content-Length Tells You When the Body Ends

Unlike the headers (which are terminated by the blank line), the body has no special terminator. The only way to know how many bytes to read is the Content-Length header.

Your server must handle three cases:

  1. No Content-Length header and no body -- typical for GET requests. After the blank line, the message is complete.
  2. Content-Length: 0 -- there is a header but the body is empty. Treat the same as case 1.
  3. Content-Length: N (where N > 0) -- read exactly N bytes from the stream after the blank line.

Putting It All Together: Parsing a Request

Here is the sequence your server should follow for each connection:

  1. Read the request line. Split it to get method, path, and version.
  2. Read headers one line at a time until you hit an empty line. Store them in a map.
  3. Check the map for Content-Length. If present and > 0, read that many bytes as the body.
  4. Route the request based on method and path.
  5. Build and send the response: status line, headers, blank line, body.
Tip: When writing your response, always include Content-Length so the client knows when the body ends. Compute the byte length of your body string (body.getBytes().length in Java) and set that as the header value.
