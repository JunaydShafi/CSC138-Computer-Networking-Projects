Debugging Your Server

Building an HTTP server from scratch means you are responsible for every byte in the response. Here are the most common mistakes students make and how to fix them.

Start Simple: Test /status First

Before testing pub-sub logic, make sure your server can handle the simplest possible request. Implement a /status endpoint that returns a plain text response. Get this working with curl before touching anything else. If /status does not work, the problem is in your HTTP formatting, not your pub-sub logic. 

curl -v http://localhost:8080/status

Common Mistakes

Bug: Using \n instead of \r\n

Symptom: curl says "Recv failure" or shows garbled output. With -v, the response headers look wrong or are missing.

Cause: HTTP/1.1 requires lines to end with \r\n (CRLF), not just \n. This is defined in RFC 9112. Java's println() only writes \n.

Fix: Write \r\n explicitly:

// WRONG
out.print("HTTP/1.1 200 OK\n");

// RIGHT
out.print("HTTP/1.1 200 OK\r\n");
Warning: This is the number one mistake. If you use PrintWriter.println() to write HTTP responses, your server will silently produce malformed HTTP on some platforms. Always use print() with explicit \r\n.

Bug: Forgetting the blank line between headers and body

Symptom: curl shows headers but no body, or treats the body as part of the headers.

Cause: HTTP requires an empty line (a bare \r\n) after the last header, before the body begins. This is how the client knows the headers are done.

Fix: Always write the blank line:

out.print("HTTP/1.1 200 OK\r\n");
out.print("Content-Type: text/plain\r\n");
out.print("Content-Length: 11\r\n");
out.print("\r\n");                        // blank line -- do not forget this
out.print("hello world");
out.flush();

Bug: Wrong Content-Length

Symptom: The response is truncated (Content-Length too small) or curl hangs waiting for more data (Content-Length too large).

Cause: Content-Length must be the exact number of bytes in the body. If you compute it from a Java String, remember that String.length() returns the number of characters, which differs from bytes for non-ASCII text.

Fix: Convert to bytes first, then measure:

String body = "hello world";
byte[] bodyBytes = body.getBytes("UTF-8");
int contentLength = bodyBytes.length;

out.print("Content-Length: " + contentLength + "\r\n");
out.print("\r\n");
outputStream.write(bodyBytes);
outputStream.flush();
Tip: For plain ASCII text, String.length() and byte length are the same. But it is a good habit to use getBytes() so you do not get bitten later.

Bug: Not flushing the output stream

Symptom: curl hangs indefinitely. The server appears to have processed the request (your debug prints show up), but the client never gets a response.

Cause: Java buffers output. If you write your response but never call flush(), the data sits in a buffer and never goes out over the network.

Fix: Call flush() after writing the complete response:

out.print("HTTP/1.1 200 OK\r\n");
out.print("Content-Length: 0\r\n");
out.print("\r\n");
out.flush();    // actually send the bytes

Bug: Reading the body when there is none

Symptom: The server hangs after reading the request line and headers. It never sends a response.

Cause: Your code tries to read a request body (e.g., calling read() for Content-Length bytes), but the request has no body. GET requests typically have no body. If Content-Length is 0 or missing, there is nothing to read.

Fix: Only read the body if Content-Length is present and greater than 0:

int contentLength = 0;
// ... parse headers, look for Content-Length ...

String body = "";
if (contentLength > 0) {
    char[] buf = new char[contentLength];
    in.read(buf, 0, contentLength);
    body = new String(buf);
}
Warning: If you call readLine() to try to read the body, it will block until it sees a newline. The body may not end with a newline. Always use read(char[], offset, length) with the exact Content-Length.

Bug: Connection hangs (general)

Symptom: curl just sits there. No response, no error. You have to Ctrl-C to kill it.

Cause: This is almost always one of two things:

  • The server is trying to read data the client never sent (see above).
  • The server wrote a response but did not flush (see above).

Fix: Add print statements to trace exactly where your server is stuck:

System.out.println("[DEBUG] Parsed request line: " + method + " " + path);
System.out.println("[DEBUG] Content-Length: " + contentLength);
System.out.println("[DEBUG] About to read body...");
// ... read body ...
System.out.println("[DEBUG] Body read. Sending response...");
// ... write response ...
System.out.println("[DEBUG] Response sent and flushed.");

Run your server, send a curl request, and look at which debug line is the last one printed. That tells you exactly where the hang is.

Using Print Statements Effectively

System.out.println() is your best debugging tool for this project. Print at these key points:

  • When a new connection is accepted (print the client address).
  • After parsing the request line (print the method and path).
  • After parsing all headers (print header count and Content-Length).
  • After reading the body (print the body length or first N characters).
  • Before sending the response (print the status code).
  • After flushing (confirm the response was sent).
Tip: Prefix debug output with [DEBUG] or a timestamp so you can distinguish it from normal output and easily remove it later.

Debugging Strategy

  1. Get /status working with curl -v.
  2. Add /subscribe. Test it. Verify the server stores the subscription.
  3. Add /publish. Test it. Use debug prints to confirm the message is queued.
  4. Add /poll. Test the full sequence: subscribe, publish, poll.
  5. Only after all endpoints work individually, test them in combination.

Do not try to build everything at once. Each endpoint is testable in isolation with curl. Build and test one at a time.
