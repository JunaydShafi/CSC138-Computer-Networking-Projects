ConcurrentHashMap and Thread-Safe Collections

When multiple virtual threads handle connections simultaneously, they share data structures like topic subscriber lists and message queues. Regular collections like HashMap are not safe for concurrent access. This tutorial covers the thread-safe alternatives you need for the pub-sub broker.

Why HashMap Is Unsafe

A HashMap is not designed for concurrent modification. If two threads call put() at the same time, the internal array can become corrupted: entries can be lost, the map can enter an infinite loop during resize, or you can get wrong values back from get(). These bugs are intermittent and extremely hard to reproduce.

// UNSAFE -- do not use in a multi-threaded server
Map<String, Set<String>> subscribers = new HashMap<>();

ConcurrentHashMap Basics

ConcurrentHashMap is a drop-in replacement that is safe for concurrent reads and writes. Individual operations like put(), get(), and remove() are atomic.

import java.util.concurrent.ConcurrentHashMap;

ConcurrentHashMap<String, String> map = new ConcurrentHashMap<>();
map.put("key", "value");
String val = map.get("key");  // "value"
map.remove("key");

computeIfAbsent: Atomic Lazy Initialization

A common pattern in the pub-sub broker: when a subscriber joins a topic, you need to get the set of subscribers for that topic, creating it if it does not exist yet. With a regular map, this requires a check-then-act sequence that is not atomic:

// UNSAFE race condition with regular HashMap
if (!map.containsKey(topic)) {
    map.put(topic, ConcurrentHashMap.newKeySet());
}
map.get(topic).add(subscriberId);

computeIfAbsent does this atomically in one call:

ConcurrentHashMap<String, Set<String>> topicSubscribers =
    new ConcurrentHashMap<>();

// Atomically get or create the subscriber set for a topic
topicSubscribers.computeIfAbsent(topic, k -> ConcurrentHashMap.newKeySet())
                .add(subscriberId);

If the key does not exist, the lambda runs to create the initial value. If another thread is simultaneously creating the same key, only one lambda executes and both threads get the same set back.

Concurrent Sets with newKeySet()

ConcurrentHashMap.newKeySet() returns a thread-safe Set. Use it wherever you need a set that multiple threads will read and modify:

Set<String> subscribers = ConcurrentHashMap.newKeySet();
subscribers.add("client-1");
subscribers.add("client-2");
subscribers.remove("client-1");

// Iterate safely -- snapshot semantics
for (String sub : subscribers) {
    // deliver message to sub
}
Tip: Iteration over a ConcurrentHashMap.newKeySet() does not throw ConcurrentModificationException. It reflects a weakly consistent view of the set, which is fine for pub-sub delivery.

ConcurrentLinkedQueue for Mailboxes

Each subscriber needs a queue of pending messages (a mailbox). ConcurrentLinkedQueue is an unbounded, thread-safe, FIFO queue. Producers add messages with add(); the consumer retrieves them with poll() (returns null if empty).

import java.util.concurrent.ConcurrentLinkedQueue;

ConcurrentLinkedQueue<String> mailbox = new ConcurrentLinkedQueue<>();

// Publisher thread adds a message
mailbox.add("{\"topic\":\"weather\",\"message\":\"sunny\"}");

// Subscriber thread retrieves messages
String msg = mailbox.poll(); // returns null if queue is empty

Pub-Sub Data Structures

Here is how these collections fit together for the broker. You need three maps:

// Topic -> set of subscriber IDs
ConcurrentHashMap<String, Set<String>> topicSubscribers =
    new ConcurrentHashMap<>();

// Subscriber ID -> queue of pending messages
ConcurrentHashMap<String, ConcurrentLinkedQueue<String>> mailboxes =
    new ConcurrentHashMap<>();

When a client subscribes to a topic:

topicSubscribers
    .computeIfAbsent(topic, k -> ConcurrentHashMap.newKeySet())
    .add(subscriberId);

mailboxes.computeIfAbsent(subscriberId, k -> new ConcurrentLinkedQueue<>());

When a client publishes a message to a topic:

Set<String> subs = topicSubscribers.get(topic);
if (subs != null) {
    for (String sub : subs) {
        ConcurrentLinkedQueue<String> q = mailboxes.get(sub);
        if (q != null) {
            q.add(message);
        }
    }
}

When a subscriber polls for messages:

ConcurrentLinkedQueue<String> q = mailboxes.get(subscriberId);
if (q != null) {
    String msg = q.poll();
    // if msg is null, no messages are waiting
}
Tip: All of these operations are safe to call from multiple threads simultaneously without any synchronized blocks. That is the entire point of using concurrent collections -- they handle the locking internally.
