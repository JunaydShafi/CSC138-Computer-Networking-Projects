Virtual Threads

Virtual threads (introduced in Java 21) let you spawn a lightweight thread per connection without worrying about thread pool sizing or resource limits. This tutorial shows how to use them in your server.

The Problem: Blocking Accept Loop

Consider this single-threaded server:

while (true) {
    Socket client = serverSocket.accept();
    handleClient(client); // blocks until this client is done
}

While handleClient is running for one client, no other client can connect. If a client is slow or sends a large request, every other client is stuck waiting. A server that handles one client at a time is not useful.

The Solution: One Thread Per Connection

Spawn a new virtual thread for each accepted connection:

while (true) {
    Socket client = serverSocket.accept();
    Thread.ofVirtual().start(() -> {
        try {
            handleClient(client);
        } finally {
            try { client.close(); } catch (IOException e) { }
        }
    });
}

That is the entire pattern. The accept loop immediately returns to waiting for the next connection, while the new virtual thread handles the current client independently.

Why Virtual Threads Instead of Platform Threads?

Platform threads (the traditional new Thread()) are backed by OS threads. Each one costs about 1 MB of stack memory and has significant creation overhead. If 10,000 clients connect simultaneously, you need 10,000 OS threads — which will exhaust memory or hit OS limits.

Virtual threads are managed by the JVM, not the OS. They are extremely cheap to create (a few hundred bytes of initial memory) and the JVM multiplexes them onto a small pool of OS threads automatically. You can have millions of virtual threads without issue.

For a server that mostly does I/O (reading from sockets, writing responses), virtual threads are ideal because the JVM can park a virtual thread during a blocking read() and reuse the underlying OS thread for another virtual thread.

Tip: You do not need an ExecutorService or a thread pool for virtual threads. The JVM handles scheduling internally. Just call Thread.ofVirtual().start() and move on.

The Lambda + Try/Finally Pattern

The lambda passed to start() is a Runnable. Use try/finally to guarantee the socket is closed even if an exception occurs:

Thread.ofVirtual().start(() -> {
    try {
        // read request, send response
        BufferedReader in = new BufferedReader(
            new InputStreamReader(client.getInputStream()));
        OutputStream out = client.getOutputStream();
        // ... handle the HTTP request ...
    } catch (IOException e) {
        // log or ignore
    } finally {
        try { client.close(); } catch (IOException e) { }
    }
});

The finally block ensures the socket is closed whether the handler completes normally, throws an exception, or the client disconnects unexpectedly.

Complete Server Skeleton

import java.net.*;
import java.io.*;

public class Server {
    public static void main(String[] args) throws IOException {
        ServerSocket serverSocket = new ServerSocket(8080);
        System.out.println("Listening on port 8080");

        while (true) {
            Socket client = serverSocket.accept();
            Thread.ofVirtual().start(() -> {
                try {
                    handleClient(client);
                } finally {
                    try { client.close(); } catch (IOException e) { }
                }
            });
        }
    }

    static void handleClient(Socket client) {
        try {
            BufferedReader in = new BufferedReader(
                new InputStreamReader(client.getInputStream()));
            OutputStream rawOut = client.getOutputStream();
            PrintWriter headerWriter = new PrintWriter(rawOut, false);

            // Read request, parse it, send response
            // ...

        } catch (IOException e) {
            // client disconnected or I/O error
        }
    }
}
Tip: Virtual threads require Java 21 or later. Make sure you are compiling and running with an appropriate JDK version.
