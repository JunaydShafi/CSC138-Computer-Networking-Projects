Testing with curl

What is curl?

curl is a command-line tool for making HTTP requests. It comes pre-installed on macOS and Linux, and is available on Windows through Git Bash or WSL. It lets you act as an HTTP client without writing any code -- perfect for testing your server one endpoint at a time.

Flag Reference

Here are the flags you will use most often:

Flag Meaning
-X METHOD Set the HTTP method (GET, POST, PUT, DELETE). GET is the default.
-H "Name: Value" Add a request header. You can use -H multiple times.
-d "data" Send a request body. Automatically sets the method to POST if -X is not given.
-v Verbose mode. Shows the full request and response, including all headers.

Simple GET Request

Start by testing your /status endpoint. This is the simplest request your server should handle: 

curl http://localhost:8080/status

Expected output (something like):

Server is running
Tip: Always test /status first. If this does not work, nothing else will. Get this endpoint working before you move on.

POST Request: Subscribing

To subscribe a client to a topic, send a POST with an X-Client-ID header: 

curl -X POST -H "X-Client-ID: alice" http://localhost:8080/subscribe/weather

This tells the broker: "alice wants to receive messages published to the weather topic." The -X POST sets the method, and -H adds the custom header that identifies the client.

POST with Body: Publishing

To publish a message, send a POST with a body using -d: 

curl -X POST -H "X-Client-ID: bob" -d "sunny today" http://localhost:8080/publish/weather

The -d "sunny today" flag sets the request body. curl will automatically include a Content-Length header matching the size of the body.

Watch out: When you use -d, curl sets Content-Type: application/x-www-form-urlencoded by default. Your server should not depend on a particular Content-Type for publish. If you want to send plain text explicitly, add -H "Content-Type: text/plain".

Verbose Mode

When something is not working, use -v to see the full HTTP exchange: 

curl -v http://localhost:8080/status

This produces output like:

* Connected to localhost (127.0.0.1) port 8080
> GET /status HTTP/1.1
> Host: localhost:8080
> User-Agent: curl/8.1.2
> Accept: */*
>
< HTTP/1.1 200 OK
< Content-Type: text/plain
< Content-Length: 17
<
Server is running

Lines starting with > are what curl sent (the request). Lines starting with < are what your server sent back (the response). This is invaluable for spotting formatting errors in your HTTP responses.

Complete Test Sequence

Here is a full end-to-end test you can run in your terminal. Execute each command in order:

Step 1: Check the server is up

curl http://localhost:8080/status

Step 2: Subscribe alice to "weather"

curl -X POST -H "X-Client-ID: alice" http://localhost:8080/subscribe/weather

Step 3: Subscribe bob to "weather"

curl -X POST -H "X-Client-ID: bob" http://localhost:8080/subscribe/weather

Step 4: Publish a message

curl -X POST -H "X-Client-ID: bob" -d "sunny today" http://localhost:8080/publish/weather

Step 5: Poll alice's mailbox

curl -H "X-Client-ID: alice" http://localhost:8080/poll/alice

You should see "sunny today" in the response body.

Step 6: Poll alice again

curl -H "X-Client-ID: alice" http://localhost:8080/poll/alice

This time the mailbox should be empty (204 No Content or an empty body, depending on your design).

Tip: You can put these commands in a shell script (e.g., test.sh) and run them all at once. Add echo statements between commands to label the output:

echo "=== subscribing alice ===" && curl -X POST ...

Troubleshooting curl

  • Connection refused: Your server is not running, or it is on a different port.
  • Hangs forever: Your server accepted the connection but never sent a response. Check that you are flushing the output stream.
  • Empty response: Your server closed the connection without writing anything. Use -v to confirm.
  • curl: (56) Recv failure: Your server crashed or closed the socket unexpectedly. Check your server's terminal for stack traces.
