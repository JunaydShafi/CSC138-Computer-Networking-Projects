GET vs POST

HTTP defines several request methods, but for the pub-sub broker you only need two: GET and POST. Understanding the difference between them -- and how your server uses the method to decide what to do -- is essential.

The Two Methods at a Glance

GET

Purpose: Retrieve information

Body: None

Content-Length: Not sent

Example: Check server status, poll for messages

POST

Purpose: Send data to the server

Body: Required

Content-Length: Must be present

Example: Publish a message to a topic

GET Requests

A GET request asks the server for information. It carries no body. Everything the server needs to fulfill the request is encoded in the URL path.

Example: Checking Server Status

GET /status HTTP/1.1\r\n Host: localhost:8080\r\n \r\n

Notice there is nothing after the blank line. The server reads the request line and headers, sees no Content-Length header, and knows the request is complete.

Example: Polling for Messages

GET /subscribe/weather HTTP/1.1\r\n Host: localhost:8080\r\n X-Client-ID: client-3\r\n \r\n

The topic name ("weather") is part of the URL path. The server extracts it by parsing the path string, not by reading a body.

POST Requests

A POST request sends data to the server. It always includes a body, and the Content-Length header tells the server how many bytes to read.

Example: Publishing a Message

POST /publish/weather HTTP/1.1\r\n Host: localhost:8080\r\n Content-Type: application/json\r\n Content-Length: 34\r\n \r\n {"temp": 68, "condition": "rainy"}

The topic is in the path (/publish/weather), and the message payload is in the body. The server must read both.

Routing: Method + Path

Your server decides what to do with a request based on the combination of the method and the path. Neither one alone is enough.

Method Path Pattern Action
GET /status Return broker status (topics, subscriber count)
GET /subscribe/{topic} Register subscriber or poll for messages
POST /publish/{topic} Publish message body to the given topic

If a client sends POST /status, that is a valid HTTP request but your server should return 405 Method Not Allowed because /status only accepts GET.

Tip: A clean way to implement routing in Java is a series of if/else checks. First match the method, then match the path or path prefix. Extract the topic name from the path by splitting on /.

Routing Logic in Pseudocode

String method = ...; // "GET" or "POST" String path = ...; // "/publish/weather" if (method.equals("GET") && path.equals("/status")) { handleStatus(); } else if (method.equals("GET") && path.startsWith("/subscribe/")) { String topic = path.substring("/subscribe/".length()); handleSubscribe(topic); } else if (method.equals("POST") && path.startsWith("/publish/")) { String topic = path.substring("/publish/".length()); handlePublish(topic, body); } else { sendError(404, "Not Found"); }

Why the Distinction Matters for Parsing

When your server reads a request, the method determines whether there will be a body to read:

  • GET -- after the blank line, stop reading. There is no body. If you try to read more bytes, your server will block waiting for data that will never arrive.
  • POST -- after the blank line, read exactly Content-Length bytes. If you read fewer, you get a truncated message. If you try to read more, your server blocks.
Tip: A safe approach is to always check for a Content-Length header after parsing all headers. If it exists and is greater than zero, read that many bytes as the body. If it does not exist, assume there is no body. This works correctly for both GET and POST without special-casing the method during body reading.

Testing with curl

You can use curl from the command line to send both types of requests to your server:

Sending a GET

curl -v http://localhost:8080/status

The -v flag shows the full request and response, including headers. This is invaluable for debugging.

Sending a POST

curl -v -X POST http://localhost:8080/publish/weather \ -H "Content-Type: application/json" \ -d '{"temp": 72, "sky": "clear"}'

The -d flag sets the body and automatically adds a Content-Length header. The -X POST sets the method (though -d implies POST by default).

Tip: Run your server in one terminal and curl in another. The verbose curl output will show you exactly what bytes are being sent and received, making it easy to spot formatting errors in your responses.
