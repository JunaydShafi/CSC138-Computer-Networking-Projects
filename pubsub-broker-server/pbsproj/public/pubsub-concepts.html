Pub-Sub Messaging

What is Publish-Subscribe?

Publish-subscribe (pub-sub) is a messaging pattern where senders of messages (publishers) do not send them directly to specific receivers (subscribers). Instead, messages are sent to a named channel called a topic, and a middleman called a broker delivers those messages to every subscriber who has registered interest in that topic.

The key insight is decoupling: the publisher does not need to know who the subscribers are, and subscribers do not need to know who published a message. The broker handles all the routing.

Topics as Named Channels

A topic is just a name -- a string like weather or sports/scores. Think of it as a radio channel. Anyone can tune in (subscribe), and anyone can broadcast (publish). The broker keeps track of which clients are listening to which topics.

Tip: In this project, topics appear as part of the URL path. For example, /subscribe/weather subscribes to the "weather" topic, and /publish/weather publishes a message to it.

How It Works

Here is the basic flow:

  1. Subscribe: A client tells the broker "I want messages from topic X."
  2. Publish: A different client sends a message to topic X.
  3. Deliver: The broker places a copy of that message into every subscriber's mailbox.
  4. Poll: Each subscriber asks the broker "Do you have any messages for me?" and retrieves them.
Publisher                        Broker                      Subscribers
=========                   ==============                  ============

                            alice subscribes
                            to "weather"
                            bob subscribes
                            to "weather"

   bob sends:
   POST /publish/weather
   Body: "sunny today"
          -------->        [Broker receives msg]
                            |
                            +---> mailbox(alice): ["sunny today"]
                            +---> mailbox(bob):   ["sunny today"]

                            alice sends:
                            GET /poll/alice
                      <--------
                            Response: "sunny today"
                            mailbox(alice): [] (now empty)

The Mailbox Model

In many pub-sub systems, the broker pushes messages to subscribers the instant they arrive. That requires holding open a connection to each subscriber, which is complex. In this project, we use a simpler mailbox model:

  • Each client has a mailbox (a queue) stored on the broker.
  • When a message is published, the broker copies it into the mailbox of every subscriber for that topic.
  • Subscribers retrieve messages by polling -- sending a request to check their mailbox.
  • Once retrieved, messages are removed from the mailbox.
Important: If a client subscribes to a topic but never polls, messages will accumulate in their mailbox. Your server should still accept and queue those messages.

Contrast: Request-Response vs Event-Driven

Normal HTTP follows a request-response pattern: the client asks for something specific, and the server responds with exactly that. Every interaction is initiated by the client, and the server only does what is asked.

Pub-sub is event-driven: a publisher creates an event ("here is a new weather reading"), and the broker distributes it to interested parties. The subscribers did not ask for that specific message at that specific time -- they expressed interest in a topic earlier, and the broker remembers.

Aspect Request-Response Pub-Sub (this project)
Who initiates? Client asks, server answers Publisher sends, broker delivers
Coupling Client knows exactly what to ask for Publisher and subscriber don't know each other
Delivery One-to-one One-to-many (fan-out)
Timing Synchronous Asynchronous (message waits in mailbox)

Why Pub-Sub Over HTTP?

In the real world, pub-sub systems use specialized protocols (MQTT, AMQP, Kafka's binary protocol). In this project, we layer pub-sub semantics on top of plain HTTP/1.1. This is not how production systems typically work, but it lets you practice HTTP parsing and server design without learning a new protocol.

The result: you get to build something that behaves like a message broker while only dealing with HTTP requests and responses that you already understand.
