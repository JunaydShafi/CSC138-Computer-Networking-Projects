HTTP Headers Reference

HTTP headers carry metadata about the request or response. They appear between the request/status line and the blank line that separates headers from the body. This page covers every header you will need for the pub-sub broker.

Header Format

Every header follows this pattern:

Header-Name: header-value\r\n

The name and value are separated by a colon and a space. The line ends with CRLF (\r\n).

Tip: When parsing, split on the first colon only. The value may contain colons (for example, Host: localhost:8080). In Java: line.split(": ", 2) will give you the name and the full value.

Case Insensitivity

HTTP header names are case-insensitive per the specification. All of the following are equivalent:

Content-Length: 42 content-length: 42 CONTENT-LENGTH: 42

Your server must handle any casing a client might send. The simplest approach is to convert header names to lowercase when storing them in your map, then always look up lowercase keys.

// Java example Map<String, String> headers = new HashMap<>(); // When parsing: String[] parts = line.split(": ", 2); headers.put(parts[0].toLowerCase(), parts[1]); // When reading: String length = headers.get("content-length");

Headers You Must Support

Header Direction Purpose
Content-Type Request and Response MIME type of the body
Content-Length Request and Response Byte length of the body
Connection Request and Response Whether to keep the connection alive or close it
Host Request The server hostname (required by HTTP/1.1)

Content-Type

Tells the receiver what format the body is in. For this project, you will almost always use JSON:

Content-Type: application/json

Your server should include this header on every response that carries a JSON body. When reading requests, you can check this header to confirm the client sent JSON, but for this project it is acceptable to assume all POST bodies are JSON.

Content-Length

The number of bytes in the body. This is how the reader knows when the body ends. You must include it on every response that has a body.

Content-Length: 28

Be careful: this is the byte count, not the character count. For ASCII text they are the same, but if you ever have multi-byte characters, use body.getBytes("UTF-8").length in Java.

Tip: A common bug is computing the Content-Length from body.length() (which returns the character count) instead of body.getBytes().length (byte count). For plain ASCII they match, but building the right habit now will save you from subtle bugs later.

Connection

Controls whether the TCP connection stays open after the response is sent.

Connection: close

For simplicity, your server can send Connection: close on every response and close the socket after writing the response. This avoids the complexity of persistent connections and request pipelining. A production server would keep the connection open by default in HTTP/1.1, but that is not required here.

Host

HTTP/1.1 requires clients to send a Host header on every request. It identifies which server the request is intended for (important when multiple domains share one IP).

Host: localhost:8080

Your server does not need to act on this header, but you should be aware that it will be present in incoming requests. Do not reject a request because it includes headers you do not recognize.

Custom Headers

The HTTP specification allows arbitrary custom headers. By convention, custom headers use an X- prefix. In the pub-sub project, you may encounter or use headers like:

X-Client-ID: subscriber-42

Custom headers are parsed the same way as standard headers. If your broker design uses a client identifier, a custom header is a clean way to carry it without polluting the URL or body.

Full Example: Request with Headers

A subscriber registering for the "news" topic:

GET /subscribe/news HTTP/1.1\r\n Host: localhost:8080\r\n X-Client-ID: client-7\r\n Connection: close\r\n \r\n

This request has no body, so there is no Content-Length or Content-Type header. The blank line at the end signals that headers are complete and there is nothing more to read.

Full Example: Response with Headers

The server confirming the subscription:

HTTP/1.1 201 Created\r\n Content-Type: application/json\r\n Content-Length: 39\r\n Connection: close\r\n \r\n {"status":"subscribed","topic":"news"}
Tip: When building your response string, a helper method that takes a status code, a body string, and returns the full HTTP response (with computed Content-Length) will prevent many bugs. Write it once, use it everywhere.
